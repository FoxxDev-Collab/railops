Multi-User Collaboration: Architecture Approaches

Three ways to build real-time crew collaboration into RailOps

A

Full Liveblocks Integration

Approach: Liveblocks rooms for everything — railroad editing, operating sessions, presence. Each railroad = a room. Session operations use nested rooms.

Pros: Richest real-time experience. Presence, cursors, conflict resolution all built-in. Fastest to build.

Cons: Vendor lock-in. Cost scales with MAU. All real-time logic lives in Liveblocks' model.

Effort: ~Medium. Liveblocks handles the hard parts.

★ Recommended

B

Hybrid: Liveblocks for Presence + Server Actions for Data

Approach: Use Liveblocks only for presence/awareness (who's online, who's editing what). All data mutations stay as server actions with optimistic updates and Liveblocks broadcast to notify other clients to refetch.

Pros: Lighter Liveblocks usage (lower cost). Data flow stays in your existing server action pattern. Easier to test. Less vendor coupling.

Cons: Slightly more latency than full Liveblocks storage. You build conflict detection yourself (optimistic locking on server actions).

Effort: ~Medium-High. More custom code but familiar patterns.

C

Pusher Channels + Custom Presence

Approach: Pusher for pub/sub events and presence channels. Build your own room/locking system. Server actions broadcast changes via Pusher, clients subscribe and update.

Pros: Cheapest at scale. Full control. No abstraction layer between you and WebSockets.

Cons: Most custom code. You build presence UI, conflict resolution, room management all yourself. Slower to ship.

Effort: ~High. Significant custom infrastructure.

Why I Recommend Approach A → B Migration Path

Start with Approach B (Hybrid) as the initial build. It keeps your existing server action patterns, uses Liveblocks only for the hard parts (presence + broadcast), and avoids putting your data model inside Liveblocks storage. If you later want richer real-time features (live cursors, simultaneous field editing), you can selectively adopt Liveblocks storage for specific features without rewriting everything.

This gives you the fastest path to a working multi-user system without over-committing to the Liveblocks data model upfront.
